# research/ — reproduction bundle

Everything needed to reproduce the paper *"The Benchmark Chooses the Winner: Operating Points, Fine-Tuning, and the Fragility of Safety-Guard Rankings"* lives in this folder.

```
research/
  notebooks/    pipeline notebooks + bundled data + (regenerated) outputs
    paper_reproduction.ipynb          canonical paper path: runs each producing script, renders tables from JSON
    smollm3_guard_reproduction.ipynb  companion demo: trains/calibrates/evaluates one guard end-to-end (not the paper path)
    data/         bundled benchmarks (incl. guard_benchmark_hard.jsonl, mortgage_split.json)
    outputs/      cached scores + summary_*.json (gitignored; regenerated locally, not committed)
    requirements.txt  pinned Python deps
    README.md
  paper/          ACM paper: benchmark_chooses_the_winner.{tex,pdf}, refs.bib, figures/, tables/
    Makefile, README.md, DRAFT.md, metrics_survey.md
  scripts/        producing scripts for the paper's experiments (see scripts/README.md)
  docs/           supporting design/results docs referenced by the notebook & paper
  .env.example
```

## Setup

```bash
cd research
python -m venv .venv && source .venv/bin/activate
pip install -r notebooks/requirements.txt
```

**Credentials (optional).** The gpt-5.4-mini baseline (OPENAI) and gated model downloads (HF) need keys; everything else runs without them. Put keys in a repo-level `research/.env` (kept *outside* the portable `notebooks/` folder) or export them in your shell — do **not** place real keys in `notebooks/.env`, because the notebook README suggests zipping `notebooks/` to share it, and a gitignored file still travels inside a zip.

```bash
cp .env.example .env   # from inside research/; then fill in keys (loaded by the scripts as a fallback)
```

## Reproduce the paper (scripts + notebook)

`paper_reproduction.ipynb` orchestrates the producing scripts for the in-house, novel/OOD, base-vs-tuned, mortgage case-study, and frontier-reground result families, and renders their tables from the emitted JSON (a `verify_cached` mode reads locally cached artifacts; a `recompute` mode reruns scoring). The **cross-family (Qwen2.5)** and **ensemble mitigation** analyses are produced by their own scripts (below), not the notebook.

```bash
cd research                          # cwd matters: script paths are relative to here
python scripts/guard_eval_pipeline.py           # in-house + novel + base-vs-tuned (per model)
python scripts/eval_mortgage_hard.py            # mortgage hardened-benchmark metrics
bash   scripts/run_extra_models.sh              # cross-family sweep (Qwen2.5 etc.)
python scripts/ensemble_probe.py                # base⊕tuned ensemble sweep (oracle/context)
python scripts/ensemble_deployable.py           # deployable per-prompt PIT ensemble
node   scripts/build_hard_jsonl.mjs             # (node) builds the superseded 318-row legacy set -> guard_benchmark_hard_legacy318.jsonl
python scripts/make_figures.py                  # regenerate paper/figures/*.pdf (values inline in the script; keep in sync with the tables by hand)
```

Mortgage case-study chain: `build_mortgage_split.py` → `train_mortgage.py` (TECHNIQUE=sft) → `eval_mortgage_tuned.py`. The paper's hardened 334-row set (`guard_benchmark_hard.jsonl`) ships in the bundle and is scored directly by `eval_mortgage_hard.py`; it is not reconstructed by a builder (`build_hard_jsonl.mjs` only rebuilds the earlier, superseded 318-row minimal-pair set).

## Build the paper

`paper/` compiles standalone (figures are committed as vector PDFs — no scripts or Python needed):

```bash
cd research/paper && make            # tectonic -> benchmark_chooses_the_winner.pdf
make figures                         # optional: regenerate figures (needs the venv above)
```

## Provenance

The in-house, novel, base-vs-tuned, mortgage, and frontier numbers trace to `notebooks/outputs/nb-smollm3-guard/*.json`, regenerated by the scripts named in `paper/README.md`; the cross-family and ensemble numbers trace to `summary_qwen2.5-1.5b.json`, `summary_ensemble_probe.json`, and `summary_ensemble_deployable.json`. These outputs are gitignored — regenerated locally, not committed — so byte-exact reruns require pinning the public benchmark dataset revisions.
