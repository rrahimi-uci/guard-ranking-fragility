# Agent Bouncer Codebase Bug Audit - 2026-07-06

## Scope

This audit reviewed the current `agent-bouncer` checkout across the FastAPI Workbench,
evaluation/ensemble pipeline, training-set builders, docs site, GitHub workflow files, and
the existing test suite.

This report is intentionally bug-focused. It does not apply source fixes.

## Verification run

Commands run from the repo root:

| Check | Result |
|---|---|
| `.venv/bin/ruff check src tests scripts` | Pass |
| `.venv/bin/pytest -q` | Pass, with one Starlette/httpx deprecation warning |
| `node docs-site/build-docs.mjs` | Pass, generated docs-site artifacts |

Audited commit: `4487ec5` (`Leaderboard: optimized small-model ensembles per objective, with
composition`).

Important caveat: the local virtualenv is Python 3.14.4, while `pyproject.toml` declares

(`pyproject.toml:10-13`). The passing local unit suite is still useful, but it is not the
same environment the project claims to support.

## Executive summary

The current unit/lint/docs checks pass. The remaining issues are mostly correctness and
validity risks: API defaults that do not match runner behavior, ensemble metrics that can be
computed over misaligned samples, silent fallback behavior in data/training paths, and missing
regression tests around those contracts.

The highest priority is to harden benchmark/evaluation validity before adding more leaderboard
features. A green test suite is not enough if a leaderboard row can be produced from stale,
misaligned, or unmerged prediction artifacts.

## Findings

### AB-001 - API default benchmark tests do not merge to the leaderboard

Severity: High  
Area: Workbench/API, evaluation artifacts

`TestConfig.benchmarks` defaults to `[]` (`src/agent_bouncer/serving/api.py:850-854`), and
`POST /api/test` only appends `--merge-scoreboard` when `cfg.benchmarks` is truthy
(`src/agent_bouncer/serving/api.py:930-943`). But the runner treats omitted benchmarks as
"run all benchmarks" (`src/agent_bouncer/training/runner.py:477-481`). That means an API
client can submit a benchmark-mode test with no explicit benchmark list, run all benchmarks,
record an eval experiment, but not merge the result into `outputs/benchmark_results.json`,
not dump prediction rows, and not trigger curve recomputation.

Impact: API and UI behavior diverge. The Workbench path usually sends explicit benchmarks, but
the public API contract makes the "default all benchmarks" run produce invisible leaderboard
results.

Fix direction: Treat `test_set is None` as benchmark mode. If benchmarks are omitted, either
pass the explicit configured benchmark list or call `run_testing.py` without `--benchmarks` but
with `--merge-scoreboard`. Add tests for both omitted `benchmarks` and explicit `benchmarks: []`.

Missing tests:

- `POST /api/test` with `{"exp": "e1"}` adds `--merge-scoreboard` and appends
  `scripts/report/compute_curves.py`.
- `POST /api/test` with `{"exp": "e1", "benchmarks": []}` has the same behavior as omitted
  benchmarks, or is rejected with a clear 400 if empty means "nothing selected".

### AB-002 - Ensemble prediction files lack sample identity and can silently misalign

Severity: High  
Area: Evaluation validity, leaderboard correctness

Prediction rows are stored as only `[gold, unsafe_pred, score, latency_ms]`
(`scripts/eval/run_benchmarks.py:155-162` and `src/agent_bouncer/training/runner.py:411-416`).
`evaluate_ensemble()` checks only that selected members share a benchmark name and have equal
row counts (`src/agent_bouncer/evaluation/ensembles.py:80-103`). It does not verify that row
`i` for every member is the same original prompt/sample.

Impact: If two guards were tested with different `per_class`, seed, cache state, leakage drops,
or benchmark refreshes but still produce the same row count, the ensemble can combine model A's
score for one sample with model B's score for another. The resulting F1/FPR/AUC can look valid
while measuring nonsense.

Fix direction: Add a stable sample identity to every prediction row, preferably normalized
text hash plus source/sample metadata. Store benchmark subset metadata (`per_class`, seed,
loader/cache version, leakage filtering hash). `evaluate_ensemble()` should reject members
unless every shared benchmark has identical sample IDs in identical order.

Missing tests:

- Same benchmark and same row count but different sample IDs must raise a `ValueError`.
- Same benchmark and same sample IDs in different order must raise, unless the implementation
  explicitly realigns by ID.
- Prediction dumps from `run_benchmarks.py` and `evaluate_and_record()` must emit the same
  schema and metadata.

### AB-003 - Ensemble optimizer scope accepts arbitrary strings and silently broadens

Severity: Medium  
Area: API validation, benchmark comparability

`EnsembleOptimizeConfig.scope` and `EnsembleOptimizeAllConfig.scope` are plain strings
(`src/agent_bouncer/serving/api.py:314-320` and `src/agent_bouncer/serving/api.py:353-357`).
`_optimize_pool()` treats only `"small"` as small-model-only; every other value becomes `None`,
which means "all prediction files" (`src/agent_bouncer/serving/api.py:322-324`).

Impact: A typo such as `"smal"` silently broadens the search to all dumped guards, including
OpenAI baselines or existing ensembles if present. That can break the intended small-model
comparison and produce a misleading optimized row.

Fix direction: Use a Pydantic `Literal["small", "all"]` or enum and reject invalid values with
422/400. Consider separate explicit switches for `include_openai_baselines` and
`include_existing_ensembles` rather than overloading `"all"`.

Missing tests:

- `POST /api/ensemble/optimize` with `scope: "smal"` returns a validation error.
- `POST /api/ensemble/optimize_all` with `scope: "all"` has a deliberate, tested inclusion policy.

### AB-004 - Curve resync is production-only, CLI-coupled, and silently swallowed

Severity: Medium  
Area: Leaderboard curves, observability

`_resync_curves()` invokes `scripts/report/compute_curves.py` via `runpy.run_path()`
(`src/agent_bouncer/serving/api.py:252-264`). It skips when `RESULTS_JSON` is monkeypatched away
from the real artifact, so unit tests do not exercise the production resync path. It also catches
`Exception` and `SystemExit`, so curve generation failures are fully silent. The target script is
CLI-shaped and parses argv inside `main()` (`scripts/report/compute_curves.py:90-95`) rather than
exposing a callable library function.

Impact: Ensemble endpoints can return success while `curves.json` and `roc_auc` are stale or
failed to rebuild. Users see a leaderboard row and assume its ROC/PR view is synchronized, but
there is no status signal when recomputation fails.

Fix direction: Refactor curve generation into a callable function with explicit `results_path`,
`pred_dir`, and `out_path` parameters. Have `_resync_curves()` call that function, log failures,
and return a sync status in ensemble endpoint responses. Keep the CLI as a thin wrapper.

Missing tests:

- Production-like resync against a temp `RESULTS_JSON` and temp prediction dir updates curves.
- Corrupt results input produces a warning/status, not a silent success.
- CLI invocation still works after refactor.

### AB-005 - `over_refusal_aware` training sets can silently lose their augmentation

Severity: Medium  
Area: Data generation, experimental validity

The `over_refusal_aware` strategy adds XSTest safe prompts inside a broad `try/except` and
swallows every exception (`src/agent_bouncer/data/training_sets.py:171-179`). If the XSTest loader
breaks, returns nothing, or changes schema, the function still writes a dataset named as
over-refusal-aware without proving that any benign-but-scary augmentation was added.

Impact: A run can be labeled as over-refusal-aware while training on the same distribution as a
plain balanced/mixed run. This corrupts experiment interpretation and makes FPR claims weaker.

Fix direction: Record augmentation metadata (`augmentation_source`, `augmentation_requested`,
`augmentation_added`) in the dataset metadata. Fail fast or emit a surfaced warning when the
strategy requires augmentation but adds zero rows.

Missing tests:

- Loader failure for XSTest is surfaced in metadata or raises a controlled error.
- `over_refusal_aware` adds `augmentation_added > 0` when safe XSTest records are available.
- Zero added rows is not silently treated as a successful over-refusal-aware dataset.

### AB-006 - Small splits and balanced subsets can produce empty eval/test sets

Severity: Medium  
Area: Data splitting, benchmark sampling

`train_test_split()` uses `int(len(uniq) * test_ratio)` (`src/agent_bouncer/data/split.py:131-138`).
`ratio_split()` does the same globally or per class (`src/agent_bouncer/data/sampling.py:101-121`).
`balanced_subset()` uses `k = min(per_class, len(safe), len(unsafe))` and returns `safe[:k] +
unsafe[:k]` (`src/agent_bouncer/evaluation/benchmarks.py:107-117`), which becomes empty if either
class is absent.

Impact: Tiny datasets can write empty holdout splits, and class-missing benchmark inputs can
create empty balanced benchmark subsets. Some paths now raise on empty benchmark/test sets, but
the split/build path can still produce invalid artifacts before evaluation.

Fix direction: Enforce minimum split cardinality when there are enough records to split, and
raise clear errors when a balanced subset cannot contain both classes. Metadata should state
when a class is missing rather than returning an empty benchmark.

Missing tests:

- One-record and two-record split behavior is defined and tested.
- Stratified split with a class count too small fails clearly or documents the fallback.
- `balanced_subset()` with only safe or only unsafe rows raises a controlled error.

### AB-007 - Training does not fail fast for missing or empty train data

Severity: Medium  
Area: Training orchestration

`train_and_record()` reads `recs = []` when `train_data` does not exist, then continues to build
the config and launch the selected trainer (`src/agent_bouncer/training/runner.py:223-250`).

Impact: The real failure happens deeper in the trainer, after temp config creation and console
banner output. That makes user-facing errors less actionable and makes it possible for a dry
mocked trainer to "succeed" with zero training examples.

Fix direction: Validate `train_data` before building the training config. Reject missing files,
empty files, and files with no valid safe/unsafe labels with clear `ValueError`s.

Missing tests:

- `train_and_record(..., train_data="missing.jsonl")` raises before trainer import/call.
- Empty JSONL raises before trainer import/call.
- All-invalid labels raise before trainer import/call.

### AB-008 - Workbench rename is incomplete in docs/generated site

Severity: Low  
Area: Branding/docs consistency

The dashboard and README now say "Agent Bouncer Workbench", and `tests/test_serve_api.py` checks
the dashboard no longer contains "Benchmark Studio". However `docs/benchmarks.md:24-25` still says
"Benchmark Workbench", and the generated `docs-site/m-benchmarks.html` contains the same stale
phrase.

Impact: The product name is inconsistent in public docs. This is not a runtime bug, but it weakens
the rename and makes generated site review harder.

Fix direction: Rename the stale docs phrase to "Agent Bouncer Workbench" and regenerate the docs
site. Add a small docs grep test for old public-facing names if the rename needs to stay enforced.

Missing tests:

- Docs grep rejects "Agent Bouncer Studio" and "Benchmark Studio" in public docs and generated site.
- Optional: reject "Benchmark Workbench" unless explicitly intended.

### AB-009 - CLI `eval` cannot evaluate trained/saved guards

Severity: Low to Medium  
Area: CLI usability, conceptual completeness

`agent-bouncer eval` always instantiates `KeywordGuard()` and has a TODO to load a trained guard
(`src/agent_bouncer/cli.py:24-30`). The Workbench and runner can evaluate trained versions, but
the CLI command named `eval` cannot.

Impact: Users who start from the CLI get a different evaluation product than the Workbench. This
is acceptable for a bootstrap demo, but it is a conceptual gap once the repo is positioned as a
train/test/compare workbench.

Fix direction: Add explicit CLI modes for `--model-path`, `--model-key`, `--technique`, or
`--saved-model`, and route benchmark-style evaluation through `training.runner.eval_only()`.
Keep `KeywordGuard` as `--guard keyword` or a documented default.

Missing tests:

- CLI eval with no model still evaluates the keyword baseline.
- CLI eval with `--path` uses the configured guard loader and records the intended model name.
- CLI eval against a saved model ID refreshes model-store metrics when requested.

## Missing test plan

Priority test additions:

1. API benchmark default tests for `/api/test` omitted/empty benchmark lists.
2. Ensemble prediction alignment tests using sample IDs and benchmark subset metadata.
3. API validation tests for optimizer `scope`.
4. Curve resync tests after refactoring `compute_curves` into a callable library path.
5. Data-builder tests for over-refusal augmentation success/failure metadata.
6. Split/sampling tests for tiny datasets and single-class balanced subsets.
7. Training fail-fast tests for missing, empty, and invalid train files.
8. Docs branding grep test for the Workbench rename.
9. CLI eval tests for trained/saved guard evaluation.

## Recommended hardening order

1. Fix AB-001 and AB-002 first. They affect whether leaderboard and ensemble results are valid.
2. Fix AB-003 and AB-004 next. They make the newly added ensemble optimizer safer and observable.
3. Fix AB-005 through AB-007 to make dataset/training failures explicit instead of silent.
4. Clean up AB-008 and AB-009 after the correctness fixes, then add the missing docs/CLI tests.

## Current release confidence

The codebase is not in a red-test state: lint, unit tests, and the docs-site build pass locally.
However, release confidence should remain limited until the evaluation artifact contracts are
hardened. The main risk is not that the app crashes; it is that the app can produce a plausible
leaderboard or ensemble result whose underlying samples/artifacts are not provably aligned.
