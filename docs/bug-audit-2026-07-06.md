## Active findings

### AB-015 - duplicate prompt alignment still shifts when one member is missing a middle occurrence

Severity: High
Status: Reproduced on current `HEAD`

Evidence:

- `src/agent_bouncer/evaluation/ensembles.py:40-52` keys duplicate rows as
  `(sample_key, occurrence_index)`.
- `src/agent_bouncer/evaluation/ensembles.py:68-74` intersects those composite
  keys across members.
- This still assumes that every member retains the same duplicate-count prefix.
  If one member is missing the second of three identical prompts, the third
  occurrence is relabeled as occurrence `1`, so later duplicates shift left.
- Direct probe:
  `[['A1', 'A2'], ['B1', 'B3']]`
  meaning member A's second duplicate was paired with member B's third duplicate
  after B skipped the middle one.
- `evaluate_cascade()` inherits the same aligner through
  `src/agent_bouncer/evaluation/ensembles.py:208`, so the problem is not
  limited to vote-style ensembles.
- Current tests cover duplicate preservation, reordered duplicates, and cascade
  happy paths (`tests/test_ensembles_eval.py:177-240`), but not the "missing
  middle duplicate" case.

Impact:

- Ensemble and cascade metrics can still silently combine the wrong duplicate
  samples when members were scored on slightly different duplicate-containing
  subsets.
- The gold-column guard catches only some corruptions; if the shifted duplicate
  has the same gold label, the metrics still look valid.

Fix direction:

- Carry a true per-example identity from the benchmark source row instead of
  inferring identity from `(normalized prompt text, local occurrence order)`.
- If a true sample id is unavailable, reject duplicate-containing keyed dumps
  rather than guessing alignment.

Missing tests:

- duplicate prompt with a missing middle occurrence is rejected or aligned by a
  true sample id
- the same protection is exercised through `evaluate_cascade()` and
  `eval_tuned()`

### AB-016 - the PDF leaderboard still drops the UI's sample-size column and mixed-size warning

Severity: Medium
Status: Reproduced on current `HEAD`

Evidence:

- The interactive leaderboard still shows the `n` column and a mixed-sample-size
  warning in `src/agent_bouncer/serving/dashboard.html:877-900`.
- The PDF builder still omits `n` entirely:
  `src/agent_bouncer/serving/leaderboard_report.py:21-33`,
  `src/agent_bouncer/serving/leaderboard_report.py:37-45`, and
  `src/agent_bouncer/serving/leaderboard_report.py:177-218`.
- Direct probe with mixed-`n` results produced:
  `{'has_n_header': False, 'mentions_sample_size_warning': False}`.

Impact:

- A PDF exported from the same scoreboard can hide the main comparability caveat
  the UI shows when rows were scored on different sample sizes.
- That makes the exported artifact easier to misread after partial benchmark
  runs or merged rows scored with different `per_class` caps.

Fix direction:

- Carry the same `n` / `n_min-n_max` aggregation into the PDF table, or at
  minimum emit the same mixed-size warning the UI already shows.

Missing tests:

- `build_html()` includes an `n` column or explicit warning when rows have
  different sample sizes

### AB-017 - `save_model` can still attach an unrelated evaluation's metrics to the wrong trained model

Severity: High
Status: Reproduced on current `HEAD`

Evidence:

- `src/agent_bouncer/serving/api.py:541-549` loads `cfg.train_exp` and
  `cfg.eval_exp` independently, then copies whatever metrics are present in
  `eval_exp` into the saved record with no linkage check.
- The current API test still covers only the happy path:
  `tests/test_serve_api.py:297-314`.
- Direct probe saved a `distilbert` SFT training record while attaching a
  `qwen3-0.6b` GRPO eval result. The stored record came out as:
  `{'saved_base_model': 'distilbert', 'saved_technique': 'sft', 'saved_metrics_f1': 0.99, 'saved_benchmark_keys': ['xstest']}`.

Impact:

- Saved-model comparisons can present the wrong benchmark metrics for the wrong
  model.
- Once persisted, the corrupted metadata looks normal to downstream
  comparison/eval-only flows, so operators get no obvious signal that the saved
  record is invalid.

Fix direction:

- Require the eval experiment to match the selected training experiment by
  model/version, or persist an explicit training-run linkage inside eval
  experiments and verify it here.
- Reject mismatches with `400` instead of silently grafting foreign metrics.

Missing tests:

- mismatched `train_exp` / `eval_exp` is rejected
- missing `eval_exp` behavior is explicitly pinned (empty metrics vs. clearer
  error), depending on intended UX

### AB-018 - one malformed JSON file can still take down the filesystem model-store paths

Severity: Medium
Status: Reproduced on current `HEAD`

Evidence:

- `src/agent_bouncer/tracking/model_store.py:115-119` blindly `json.load()`s
  the file for `get()`.
- `src/agent_bouncer/tracking/model_store.py:140-145` does the same for every
  `*.json` file during `list()`.
- `src/agent_bouncer/serving/api.py:506-517` exposes those calls directly
  through `/api/saved_models` and `/api/saved_models/{model_id}`.
- Direct probe with a single malformed `broken.json` file caused
  `JSONDecodeError Expecting property name enclosed in double quotes...`.
- Current tests only assert that non-JSON files are ignored:
  `tests/test_model_store.py:100-105`.

Impact:

- A single malformed or partially copied JSON file can break the saved-models
  API surface.
- This is especially brittle because the filesystem backend is still the default
  store backend.

Fix direction:

- Treat malformed JSON like any other unreadable artifact: skip it during
  listing, and return `None` / `404` on `get()` instead of crashing the request.
- Log or quarantine the bad file so operators can see what broke.

Missing tests:

- malformed JSON files are skipped by `list()`
- malformed record files do not crash `get()` or `/api/saved_models/{id}`

### AB-019 - the committed cascade endpoint still ignores a partially specified stage selection

Severity: Medium
Status: Reproduced on current `HEAD`

Evidence:

- `src/agent_bouncer/serving/api.py:413-418` defines `stage1` and `stage2` as
  optional request fields.
- `src/agent_bouncer/serving/api.py:434-442` honors caller input only when
  both fields are present; otherwise it falls through to `optimize_cascade()`
  and auto-picks both stages.
- The current cascade API tests cover scope validation and the all-auto happy
  path only:
  `tests/test_serve_api.py:436-469`.
- Direct probe calling `build_cascade()` with `stage1='user-gate'` and
  `stage2=None` returned:
  `{'stage1': 'auto-gate', 'stage2': 'auto-filter', ...}`,
  which proves the supplied stage was ignored.

Impact:

- API callers cannot pin one cascade stage and auto-pick the other, even though
  the schema/docstring imply that omitted stages are the only ones auto-filled.
- The endpoint can persist a cascade composition different from what the caller
  requested.

Fix direction:

- Respect any provided stage and auto-pick only the missing peer, or reject
  partial specification with `400` if that mode is intentionally unsupported.

Missing tests:

- `stage1` provided and `stage2` omitted
- `stage2` provided and `stage1` omitted

### AB-020 - cascade AUC is contaminated by filter scores for rows the gate never forwards

Severity: Medium
Status: Reproduced on current `HEAD`

Evidence:

- `src/agent_bouncer/evaluation/ensembles.py:215-223` says non-gated inputs
  "short-circuit" to the gate score, but it actually calls `combine(..., "intersection")`
  for every aligned pair.
- `src/agent_bouncer/models/ensemble.py:43-44` defines `"intersection"` score as
  `min(scores)`, so the filter's score still influences the stored cascade score
  even when `gate_flags` is false and the filter should not run at all.
- Direct probe with a gate that never flagged anything produced:
  `auc_low 0.5`
  and
  `auc_high 1.0`
  solely by changing the filter's hypothetical scores on the same non-forwarded
  rows.

Impact:

- The cascade's reported `roc_auc` can change even when the deployed cascade
  behavior is identical.
- This distorts leaderboard ordering and PDF/report interpretation for cascades,
  because AUC is now partly driven by scores from a model that never executed on
  those samples.

Fix direction:

- When `gate_flags` is false, set the cascade score from the gate alone and do
  not incorporate the filter score at all.
- Alternatively compute AUC from an explicit cascade scoring rule rather than
  reusing generic intersection scoring.

Missing tests:

- changing the filter scores on gate-safe rows does not change cascade AUC
- gate-never-flags and gate-always-flags edge cases for cascade scoring

### AB-021 - `optimize_cascade()` can choose non-overlapping "best" stages and fail even when a valid pair exists

Severity: Medium
Status: Reproduced on current `HEAD`

Evidence:

- `src/agent_bouncer/evaluation/ensembles.py:240-244` picks the highest-recall
  model and highest-precision different model independently, then calls
  `evaluate_cascade()` on that single pair.
- `src/agent_bouncer/evaluation/ensembles.py:202-204` rejects pairs with no
  shared benchmark.
- Direct probe with three models:
  - `recall_only_x` (highest recall, benchmark `x` only)
  - `precision_only_y` (highest precision, benchmark `y` only)
  - `usable_filter_x` (valid filter on benchmark `x`)
  raised:
  `ValueError the gate and filter share no common benchmark`
  even though `recall_only_x -> usable_filter_x` is a valid cascade.
- The current cascade tests cover only the one-benchmark happy path:
  `tests/test_ensembles_eval.py:228-240`.

Impact:

- The Workbench's auto-cascade action can fail on partially tested model pools
  even when there is a scorable cascade in the available predictions.
- This makes the feature brittle in realistic workflows where benchmark coverage
  differs across models.

Fix direction:

- Rank candidate gate/filter pairs that actually share scorable benchmarks,
  instead of picking each stage independently first.
- If no full-pool pair overlaps, return a targeted error that explains which
  models are missing common benchmark coverage.

Missing tests:

- auto-cascade skips non-overlapping top models and picks a valid overlapping
  pair when one exists
- auto-cascade error message is specific when no pair shares benchmarks
