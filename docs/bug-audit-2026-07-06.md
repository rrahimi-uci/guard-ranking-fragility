## Active findings (original report, now resolved — see Resolution above)

### AB-010 - `dump_predictions.py` still emits the legacy 4-column schema

Severity: High
Status: Reproduced on current checkout

Evidence:

- `scripts/eval/dump_predictions.py:7-8` still documents `[[y, u, s, ms], ...]` and says guards
  "align by index".
- `scripts/eval/dump_predictions.py:93-98` still writes 4-element rows with no `sample_key`.
- `src/agent_bouncer/evaluation/ensembles.py:50-57` only uses keyed alignment when every row has
  a 5th element; otherwise it falls back to positional handling.
- `docs/ensembles.md:9-13` still tells users the output schema is
  `[[gold, pred, score, latency_ms], ...]`.
- Direct probe against a temp benchmark cache and `keyword-baseline` wrote `{'demo': [4]}`: the
  manual dump path still produces only 4 columns.

Impact:

- Anyone following the documented CLI ensemble workflow can still generate prediction artifacts
  that bypass the keyed-alignment fix.
- That reopens order-dependent corruption on the manual ensemble path even though the main
  benchmark runner now writes keyed rows.

Fix direction:

- Make `scripts/eval/dump_predictions.py` reuse the same shared prediction-row writer used by
  `runner.py` / `run_benchmarks.py`.
- Update every schema contract comment/doc from 4 columns to 5 columns.

Missing tests:

- `dump_predictions.py` integration test that asserts 5-element rows with a stable `sample_key`
- regression test that the documented CLI ensemble workflow produces keyed dumps end-to-end

### AB-011 - `eval_tuned()` still aligns members by raw row index

Severity: High
Status: Reproduced on current checkout

Evidence:

- `scripts/eval/eval_ensembles.py:76-88` builds its validation/test records with `rows[m][i]` and
  never calls the keyed alignment logic from `src/agent_bouncer/evaluation/ensembles.py:40-56`.
- There is no current test coverage for `eval_tuned()` (`rg` over `tests/` finds none).
- Direct probe with two keyed members containing the same samples in different order produced:
  - shared evaluator: `{'f1': 1.0, 'fpr': 0.0, 'n': 4}`
  - tuned path: `{'threshold': 0.4, 'f1': 0.666..., 'fpr': 1.0, 'n': 2}`

Impact:

- The `ensemble-tuned` result can still be materially wrong when member prediction dumps differ in
  order or subset.
- This is exactly the corruption class the AB-003 fix was supposed to eliminate, but it remains on
  the tuned-ensemble path.

Fix direction:

- Align keyed rows once, up front, through shared library code before any val/test split or
  threshold tuning.
- Reject legacy dumps with mismatched gold columns or counts instead of silently indexing across
  them.

Missing tests:

- reordered keyed members keep identical tuned metrics
- keyed members with partially overlapping subsets tune/score on the keyed intersection only

### AB-012 - keyed alignment silently drops duplicate prompts

Severity: High
Status: Reproduced on current checkout

Evidence:

- `src/agent_bouncer/evaluation/ensembles.py:31-37` defines `sample_key()` as a hash of normalized
  prompt text only.
- `src/agent_bouncer/evaluation/ensembles.py:50-56` converts each member into `{sample_key: row}`,
  which drops multiplicity before alignment.
- Direct probe with two guards that each had two rows for the same prompt text produced
  `{'n': 1, 'f1': 0.0}` instead of scoring both rows.

Impact:

- Duplicate prompts inside a benchmark silently disappear from ensemble scoring.
- If the duplicates have different labels, responses, or surrounding context, the resulting
  metrics are not just incomplete; they are wrong.

Fix direction:

- Use a real per-example identity from the benchmark source (row id / cache index / dataset key),
  not a text-only hash.
- If duplicates are still legal, align multisets rather than plain dicts so duplicate rows are
  preserved or explicitly rejected.

Missing tests:

- duplicate prompt texts are preserved as distinct rows when identities are unique
- same-text duplicates without a unique id are rejected explicitly rather than silently collapsed

### AB-013 - `ratio_split(..., stratified=True)` can still return an empty test set

Severity: Medium
Status: Reproduced on current checkout

Evidence:

- `src/agent_bouncer/data/sampling.py:110-116` applies `holdout_count()` independently per class.
- With one `safe` and one `unsafe` example and `test_ratio=0.2`, the direct probe returned
  `{'n_train': 2, 'n_test': 0, ...}`.
- `tests/test_split.py:17-27` covers the tiny-data contract only for `holdout_count()` and
  non-stratified `train_test_split()`, not for stratified `ratio_split()`.

Impact:

- The API/UI can still create unusable stratified holdouts on tiny datasets even though the earlier
  report marked the tiny-split family as fixed.
- This is especially likely during exploratory Workbench usage, where users create very small
  filtered datasets.

Fix direction:

- Choose the global test size first, then apportion it across classes while preserving the
  "at least one total holdout row when splittable" contract.
- Add the same guarantee to `sample_and_split(..., sampling="stratified", split="ratio")`.

Missing tests:

- `ratio_split(..., stratified=True)` returns `1 train / 1 test` for one-safe-plus-one-unsafe
- `sample_and_split(... stratified+ratio ...)` preserves the same non-empty holdout contract

### AB-014 - `train_and_record()` still does not validate training-row schema

Severity: High
Status: Reproduced on current checkout

Evidence:

- `src/agent_bouncer/training/runner.py:234-246` only checks that the file exists and is non-empty.
- `src/agent_bouncer/data/loaders.py:319-321` parses JSONL rows but does not validate required
  fields.
- `src/agent_bouncer/training/runner.py:77-80` treats anything not explicitly labeled `"unsafe"` /
  `"1"` / `"true"` as safe in the banner counts.
- `tests/test_runner_train.py:65-85` only covers missing and empty files, not malformed rows.
- Direct probe with `{"text":"missing-label"}` printed a normal-looking banner (`1 safe / 0 unsafe`)
  and recorded a successful training run.
- Direct probe with `{"label":"safe"}` also passed through `train_and_record()` itself when
  downstream trainer validation was stubbed, showing that the method never enforces its own input
  contract.

Impact:

- Malformed datasets can be miscounted, mislabeled, and potentially recorded as successful
  training runs before any trainer-level failure happens.
- Experiment metadata becomes unreliable, and operator trust in the Workbench training flow drops.

Fix direction:

- Validate every row before the banner/trainer: require non-empty `text` and label in
  `{"safe","unsafe"}`.
- Fail before printing success-looking progress or recording an experiment.
- Consider a shared record schema validator reused by dataset creation, training, and evaluation
  paths.

Missing tests:

- reject row missing `label`
- reject row missing `text`
- reject unknown label values
